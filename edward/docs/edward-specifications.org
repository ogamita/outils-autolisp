* But et périmètre

=edward= est un outil en Common Lisp, sous-projet de =outils-autolisp=
(au même titre que [[file:../../dwg-identifier/][dwg-identifier]]), bâti sur les bibliothèques
=clautolisp= (=clautolisp/drawing= et =clautolisp/drawing-dwg=,
vendorisées sous =third-party/clautolisp=). Il lit et écrit des dessins
DWG/DXF *sans AutoLISP ni CAO* (DXF nativement, DWG via libredwg).

Les applications EPURE — *SCHMS* (SCHMS+), *SCHME*, *SCHMIEUX*, *PV* —
stockent leurs données métier *dans* le dessin =.dwg=, en plus des
entités graphiques :

- en *xdata* attachées aux entités, sous un /appid/ propre à
  l'application ;
- en *XRECORD* rangés dans des *dictionnaires nommés* sous le
  dictionnaire d'objets nommés (NOD), pour les données de niveau
  dessin.

=edward= sait extraire ces données, les *vider en JSON* (version 1),
puis *sélectionner et transférer* des objets applicatifs d'un dessin
source vers un dessin destination (version 2 : fusion, remplacement,
suppression, avec ou sans les entités attachées).

=dwg-identifier= /identifie/ l'application productrice (table APPID) ;
=edward= /extrait et manipule/ ce que cette application a stocké. Les
deux partagent la même pile clautolisp et les mêmes conventions de
sous-projet.

** Ce que edward N'est PAS

- Ce n'est pas un moteur CAO : pas de régénération de blocs, pas de
  rendu, pas de logique métier SCHMS/PV.
- Ce n'est pas un validateur de schéma métier : il restitue fidèlement
  ce qui est stocké, y compris des données obsolètes ou incohérentes.
- Il ne migre pas les schémas (cf. =mise_a_jour.lsp= côté SCHMS) ; il
  /signale/ la version de schéma rencontrée mais ne la convertit pas.

* Modèle de données stocké par les applications EPURE

Synthèse de l'analyse des sources =~/works/sncf-reseau/src/{schms,schme,schmieux,pv}=
et de la bibliothèque partagée EPURELIB, recoupée par =dwg-identifier=
sur des dessins réels.

** Marqueur commun : la table APPID

Chaque application enregistre une famille d'/appids/ éponyme (table
APPID) et marque ses objets par des xdata sous l'un de ces appids.

| Application | Famille d'appids                                    | Appid porteur de xdata |
|-------------+-----------------------------------------------------+------------------------|
| SCHMS       | =SCHMS=, =SCHMSPLUS=, =SCHMS_{POSTES,VOIES,LIGNES}=  | =SCHMSPLUS=            |
| SCHME       | =SCHME=, =SCHME+=, =SCHMEPLUS=                       | (à confirmer, cf. §2.5)|
| SCHMIEUX    | =SCHMIEUX=, =SCHMIEUX_*=                             | (à confirmer, cf. §2.5)|
| PV          | =PV=, =PVPLUS=, =PV2010=, =PV-SUITE=, =PVSX_*=       | =PV=                   |
| EPURE       | =EPURE=, =EPURELIB=, =EPURE_*=, =SNCF-Com_*=         | (ombrelle partagée)    |

** SCHMS / SCHMS+ — modèle riche piloté par schéma (EPURELIB)

C'est le modèle le plus structuré, et la /référence/ pour edward. Code
de référence : =schms/src-vlx/xdata_codec.lsp=,
=schms/src-vlx/dictionnaires.lsp=, =schms/src-vlx/tables.lsp=, et le
modèle de classes (=classe.lsp=, =schema.lsp=, =champ.lsp=,
=typechamp.lsp=, =instance.lsp=, =multimeta.lsp=).

*** Grammaire xdata d'une instance (sur entité)

Appid unique : =SCHMSPLUS= (=*schms_app*= dans =xdata_codec.lsp=).
=instances->xdata= sérialise via =instances->dxf= avec les codes
=1000 1005 1070 1071 1040 1002= :

#+begin_example
(-3
 ("SCHMSPLUS"                      ; appid (porté par le code -3 de entget)
  (1000 . <nom-de-classe>)         ; nom de classe
  ;; puis, par instance :
  (1070 . <version-de-schema>)     ; version du schéma de l'instance
  (1000 . <nom-de-champ>) <valeur> ; champ + valeur, répété
  ...))
#+end_example

Types de champ (=typechamp=) et codage de la valeur :

| Type   | Code DXF (xdata) | Valeur                                  |
|--------+------------------+-----------------------------------------|
| STR    | 1000             | chaîne                                  |
| INT    | 1070 / 1071      | entier (1070 si -32768..32767, sinon 1071) |
| REAL   | 1040             | réel                                    |
| BOOL   | 1070             | 0 (faux) / 1 (vrai)                      |
| ENUM   | 1000             | valeur d'énumération (chaîne)           |
| HANDLE | 1005             | handle d'entité (chaîne hexa)           |
| LIST   | 1002 { … }       | liste homogène                          |
| ALIST  | 1002 { … }       | liste d'association (schéma)            |
| OBJECT | 1002 { … }       | instance imbriquée (classe + version)   |
| nil    | 1002 "{" "}"     | absence de valeur (paires vides)        |

Les listes/objets imbriqués sont encadrés par =(1002 . "{")= …
=(1002 . "}")=. Seuls les champs non nuls sont sérialisés.

*** Forme XRECORD (stockage de niveau dessin)

=instances->xrecord= produit la *même* charge utile que ci-dessus mais
avec les codes XRECORD =1 320 60 90 210 102= (au lieu de
=1000 1005 1070 1071 1040 1002=), préfixée par
=(0 . "XRECORD") (100 . "AcDbXrecord")=.

Ces XRECORD sont rangés dans des *dictionnaires nommés* sous le NOD.
Tables de niveau dessin connues (=tables.lsp=) :

| Clé    | Dictionnaire NOD | Classe                |
|--------+------------------+-----------------------|
| LIGNES | =SCHMS_LIGNES=   | =/es/schms/bd/ligne=  |
| VOIES  | =SCHMS_VOIES=    | =/es/schms/bd/voie=   |
| POSTES | =SCHMS_POSTES=   | =/es/schms/bd/poste=  |

Structure d'un dictionnaire : entrées =(3 . <clé>)= →
=(350 . <handle-xrecord>)= ; le XRECORD pointé est désérialisé par
=xrecord->instances=. Plusieurs instances peuvent cohabiter dans un même
XRECORD (séquence version/champs répétée). L'ordre est préservé par des
métadonnées de dictionnaire triable (=schms_dictionnaire_triable=).

*** Modèle de classes et de schémas

Modèle interne (=classe.lsp=, =schema.lsp=, =champ.lsp=) :

- *Classe* (=creer_classe=) : =NOM= (ex. =/es/schms/sx2/gare=),
  =NUM_SCHEMA_COURANT=, =SCHEMAS= (alist =version → schema=),
  =NOM_DAFFICHAGE=, =CATEGORIES=.
- *Schéma* (=creer_schema=) : =DESCRIPTION= + =CHAMPS= (alist
  =nom → champ=, dans l'ordre).
- *Champ* (=creer_champ=) : =(nom . typechamp)=.

*Source des schémas : les fichiers =*_ATTR.LSP=* (≈ 206 classes :
=SX1/=, =SX2/=, =BD/=, =COMMUT/=, =COMMUN/=, …). edward *reproduit* ce
format. Une classe est une S-expression alist :

#+begin_src lisp
(
 (NOM_DAFFICHAGE . "Voie")
 (CATEGORIES "Non graphique")
 (SCHEMAS
   (3 "Ajout du type de voie"                 ; version la plus récente d'abord
     ("UUID" chaine)
     ("TAGVOIE" chaine)
     ("TYPE_VOIE" enum "PRINCIPALE" "LIAISON" "SERVICE") ; nullable
     (@inclure "/es/schms/commun/etats" 1))   ; inclusion par valeur, version figée
   (2 "Ajout de l'UUID"  ("UUID" chaine) ("TAGVOIE" chaine) ...)
   (1 "Version initiale" ("TAGVOIE" chaine) ...)))
#+end_src

Règles de parsing (=schms_lire_classe= / =conversion_type=) :

- =SCHEMAS= liste *toutes les versions* d'une classe, en général de la
  plus récente à la plus ancienne ; =NUM_SCHEMA_COURANT= = numéro
  maximum. Une version est =(N "description" champ…)=.
- Un champ est =("NOM" type args…)=. Symboles de type → typechamp :
  =chaine→STR=, =entier→INT=, =reel→REAL=, =booleen→BOOL=,
  =handle→HANDLE=, =enum→ENUM(alternatives)=, =liste→LIST(type)=,
  =objet→OBJECT(classe version)=. (Symboles AutoLISP, insensibles à la
  casse.)
- =(@inclure "/chemin/classe" version)= : *inclusion par valeur* — les
  champs de la classe référencée, à la *version indiquée* (pas la
  courante), sont aplatis (=flatmap=) à la place de la directive.
  Récursif. Ex. =gare= inclut =etats= v1 → champs =REFLECTIF=,
  =LUMINEUX=, =NON_ECLAIRE=, =FIXE=, =BAS=.
- *Résolution chemin → fichier* (=schms_nomfichier_schemas=) :
  =/es/schms/…= → substitution =/schms/→/schmsplus/=, suffixe
  =_ATTR.LSP=, puis chemin d'installation (=com_way=). Le segment de
  répertoire est en majuscules (=sx2→SX2/=, =commun→COMMUN/=,
  =bd→BD/=), le dernier segment donne le fichier (=gare→GARE_ATTR.LSP=).
  edward prend une *racine de schémas* configurable (=--schema-root=,
  par défaut l'arbre =schms/= ou une copie embarquée).
- *Encodage des =*_ATTR.LSP== : *cp1252 / CRLF* (comme les autres sources
  SCHMS+ ; les accents y sont en Windows-1252). Le lecteur de schémas
  d'edward doit décoder cp1252, sinon les =DESCRIPTION=/=NOM_DAFFICHAGE=
  accentués sont corrompus.

*Pas de migration de schéma dans edward.* SCHMS migre à la lecture
(=schms_maj_metadonnees=, =mise_a_jour.lsp=) ; edward *ne migre pas*. Il
lit chaque instance *à la version où elle est stockée*, l'interprète avec
le schéma de *cette* version, et préserve cette version (cf. §5,
translation horizontale).

*** Attachement et liens

- Instances graphiques : portées par des entités =INSERT= sélectionnées
  via =(ssget "X" '((0 . "INSERT") (-3 ("SCHMSPLUS"))))=. Accès xdata par
  =(entget ename '("SCHMSPLUS"))=, code =-3=.
- Instances de niveau dessin : XRECORD dans =SCHMS_LIGNES/VOIES/POSTES=.
- Liens : pas de référence de handle explicite généralisée ; le couplage
  passe par des champs =UUID= et par l'identité du bloc. Certains champs
  sont de type =HANDLE= (1005) et référencent une entité.

** PV — modèle plat, hérité (PV2010)

Référence : =pv/arx/PV2010/PvXDATA.lsp=, =pv/arx/PV2010/pv2010.dcl=.

- Appid unique : =PV= (=(regapp "PV")=). Les noms =PVPLUS/PV2010/PV-SUITE/PVSX=
  sont des éditions, pas des appids porteurs de xdata.
- *Pas de dictionnaire, pas de XRECORD* : toute la donnée est en xdata
  sur des entités graphiques.
- *Pas de nom de classe ni de version de schéma* : 5 champs *chaîne*
  fixes, codés =1000=, encadrés par =(1002 . "{")= … =(1002 . "}")= :

#+begin_example
(-3 ("PV" (1002 . "{")
          (1000 . <Désignation>)
          (1000 . <Symbole>)
          (1000 . <Document Référence>)
          (1000 . <Commentaire(1)>)
          (1000 . <Commentaire(2)>)
          (1002 . "}")))
#+end_example

- Lecture : =(cdr (assoc -3 (entget ename '("PV"))))= ; le code descend
  aussi dans les définitions de blocs pour trouver les entités porteuses.
- Une couche =XDATA (ne pas modifier)= marque les entités concernées.

** SCHME et SCHMIEUX — appids confirmés, codec à confirmer

=dwg-identifier= confirme, sur dessins réels, que SCHME et SCHMIEUX
*enregistrent bien leurs appids* (=SCHME/SCHME+/SCHMEPLUS=,
=SCHMIEUX/SCHMIEUX_*=). En revanche, l'exploration des sources n'a *pas
localisé* de codec xdata/dictionnaire équivalent à celui de SCHMS : ces
applications semblent surtout porter leur état dans la *géométrie, les
blocs, les attributs de blocs et les calques*, et possiblement des xdata
de forme plus simple.

*Décision.* Pour SCHME et SCHMIEUX, edward s'en tient à la *couche brute*
(§3) pour l'instant : elle vide /toute/ xdata sous /n'importe quel/ appid
EPURE et /tout/ XRECORD de /n'importe quel/ dictionnaire nommé,
indépendamment du codec applicatif. Un décodeur spécifique
SCHME/SCHMIEUX sera *affiné plus tard*, sur la base d'un relevé de
dessins réels révélant une grammaire stable (cf. §8). L'effort de
décodage riche porte d'abord, et pleinement, sur *SCHMS*.

** Données hors xdata/XRECORD (toutes applications)

Blocs et définitions de blocs, attributs de blocs, calques/linetypes,
variables d'en-tête, dictionnaires non applicatifs : ce sont des données
DWG standard. edward les restitue au titre du *vidage des entités*
(§4.3), sans interprétation métier. Exports externes (DBF, JSON métier)
sont hors dessin et hors périmètre.

* Architecture

Deux couches, pour être robuste y compris là où le codec applicatif est
inconnu :

1. *Couche générique (brute), agnostique de l'application.* À partir
   d'un =drawing= clautolisp :
   - énumère la table APPID (réutilise la logique de =dwg-identifier= /
     factorisable) ;
   - pour chaque entité, extrait /toutes/ les xdata, groupées par appid,
     en *paires de codes DXF brutes* =(code . valeur)= ;
   - parcourt le NOD et tous les dictionnaires nommés, restitue chaque
     XRECORD en paires DXF brutes ;
   - restitue les entités (type, handle, calque, bloc, propriétés DXF).
   Cette couche ne /comprend/ rien : elle transcrit fidèlement.

2. *Décodeurs spécifiques par application (optionnels, enrichissants).*
   Au-dessus de la couche brute, des décodeurs reconstruisent des
   /instances typées/ quand la grammaire est connue :
   - =decode-schms= : grammaire EPURELIB (classe, version, champs typés,
     imbrication =1002 {}=, XRECORD des dictionnaires =SCHMS_*=). Décodage
     /auto-descriptif/ possible (chaque champ est =(1000 . nom) valeur=)
     même sans charger les =*_ATTR.LSP= ; le chargement des schémas
     n'est nécessaire que pour /nommer/ les types et valider — optionnel,
     phase ultérieure.
   - =decode-pv= : 5 champs chaîne fixes.
   - =decode-schme= / =decode-schmieux= : à définir (cf. §8).
   Un décodeur produit un objet structuré ; à défaut, la couche brute
   reste disponible (=--raw=).

Le registre de décodeurs associe =appid → décodeur= ; les appids inconnus
retombent sur la sortie brute, /jamais/ une perte de donnée.

** Principe directeur : informé par le schéma, fidèle au brut

Les =*_ATTR.LSP= décrivent les objets *du point de vue de l'application*.
Mais edward traite des =.dwg= dans un *format de plus bas niveau* qui
*peut diverger* de cette spécification (xdata écrites par une version
plus ancienne, objets corrompus, champs en trop ou manquants, types
inattendus, version absente du schéma). C'est précisément pourquoi un
outil de manipulation de ces données est nécessaire.

Règle absolue : *le DWG fait foi*. Le schéma /informe/ — il sert à
*nommer, valider, afficher et rechercher* — mais ne /filtre/ jamais. Tout
décodage SCHMS produit donc, conjointement :

1. la *forme brute* =raw= (paires DXF), toujours présente et exhaustive ;
2. une *interprétation* selon le schéma de la classe + version lue
   (champs nommés et typés) /quand elle s'applique/ ;
3. des *annotations de divergence* explicites, jamais silencieuses :
   classe inconnue, version absente des =SCHEMAS=, champ hors schéma,
   incohérence de type, arité inattendue, structure =1002 {}= mal formée.

Ainsi un objet conforme est richement décodé ; un objet divergent reste
*dumpé intégralement* (brut + ce qui a pu être interprété + le diagnostic
de l'écart). Aucune donnée n'est perdue ni « corrigée » à la lecture.

** Dépendance clautolisp — capacités et lacunes

API de *lecture* (confirmée par =dwg-identifier= et l'API clautolisp) :
=read-drawing=, =map-entities=/=do-entities=, =find-entity=,
=entity-dxf= (liste de paires =(code . valeur)=), =entity-kind=,
=entity-handle-string=, =map-table-records=/=find-table-record=,
=symbol-table-record-name=/=-data=, =drawing-dictionary=,
=map-dictionary=/=dictionary-get=, =find-object= (XRECORD & objets non
graphiques par handle), =drawing-entity-count=, =drawing-format=,
variables d'en-tête, blocs (=map-blocks=, =block-entities=).

API d'*écriture* (pour la v2) : =write-drawing= (DXF ; DWG via
=drawing-dwg= + libredwg), =copy-drawing= (copie profonde),
=allocate-handle=, =add-entity=/=modify-entity=/=set-entity-deleted-status=,
=add-table-record=, =dictionary-put=/=dictionary-remove=, =add-object=,
=add-block=, =set-drawing-variable=.

*Lacunes connues à combler dans edward* :
- *Pas de mutateur xdata de haut niveau* : pour écrire/copier des xdata,
  edward construit lui-même la liste DXF =1000…1071= et appelle
  =modify-entity=.
- *Pas de constructeur XRECORD de haut niveau* : edward assemble
  =((0 . "XRECORD") (5 . handle) (330 . owner) …)= et appelle
  =add-object=, puis chaîne dans le dictionnaire via =dictionary-put=.
- *Pas de résolution de handle générique* : edward fournit
  =resolve-handle = find-entity ⊕ find-object=.

Note : les signatures exactes seront /vérifiées contre la source
clautolisp/ au moment de l'implémentation (la couche d'écriture est la
plus récente et la moins stabilisée). edward isole cette dépendance
derrière un petit module d'adaptation (=dwg-access=) pour absorber les
écarts.

* Version 1 — vidage en JSON

** Objectif

Lire un (ou plusieurs) dessin(s) et émettre un *document JSON* décrivant
(a) les données applicatives stockées, étiquetées par application, et
(b) les entités du dessin avec leur type et leurs propriétés.

** Organisation du JSON : par application, puis liens vers les entités

Conformément à la demande : *trier par application* plutôt que par
entité. Une donnée applicative attachée à une entité référence l'entité
par son handle ; la section =entities= porte le détail des entités une
seule fois.

#+begin_src json
{
  "edward": "1.0.0",
  "source": "N1A_V1.dwg",
  "format": "dwg",
  "drawing": {
    "version": "AC1027",
    "codepage": "ANSI_1252",
    "entity_count": 37329,
    "appids": ["ACAD", "SCHMS", "SCHMSPLUS", "SNCF-Com_Echelle", "..."]
  },
  "applications": [
    {
      "application": "SCHMS",
      "label": "SCHMS+ (EPURE)",
      "appids": ["SCHMS", "SCHMSPLUS"],
      "decoder": "schms",
      "entity_data": [
        {
          "entity": "1A2B",                 // handle, renvoie à "entities"
          "appid": "SCHMSPLUS",
          "class": "/es/schms/sx2/gare",
          "schema_version": 3,
          "schema_status": "ok",            // ok | unknown-class | unknown-version
                                            //  | divergent (cf. divergences)

      "handle": "1A2B",
      "type": "INSERT",

                      "name": "GARE" },

    }
  ]

Principes :
- *Pas de perte* : chaque donnée applicative porte sa forme =raw= (paires
  DXF) en plus, le cas échéant, de sa forme décodée =fields=. Les appids
  sans décodeur ne portent que =raw=.
- *Étiquetage application* : chaque bloc =applications[]= regroupe ce
  qu'une application a stocké (xdata d'entités + données de dessin).
- *Liens entités* : =entity_data[].entity= et =entities[].handle= se
  correspondent ; =entities[].applications= liste réciproquement les
  applications présentes sur l'entité (cas du dessin multi-applications).
- *Multi-applications* : un même dessin peut produire plusieurs blocs
  =applications[]= ; une même entité peut figurer dans plusieurs d'entre
  eux (xdata de plusieurs appids).

** Options de vidage

- Filtrer par application (=--app schms,pv=) ou par appid.
- =--raw= : n'émettre que la forme brute (pas de décodage), utile pour
  diagnostiquer ou pour les applications sans décodeur.
- =--no-entities= : omettre la section =entities= (données applicatives
  seules) ; =--entities-only= pour l'inverse.
- =--pretty= / =--compact= : indentation.
- Sélection d'entités par type, calque, ou plage de handles (réutilisée
  en v2 pour la sélection).

** Cas de validation V1 — fidélité du round-trip

Avant toute écriture sémantique (v2), il faut *de-risquer* la chaîne
clautolisp + libredwg : garantit-elle un /round-trip/ sans perte ? C'est
le *test d'acceptation de la v1* (et le prérequis du §5.2).

Principe : lire un dessin réel, le réécrire à l'identique (passthrough,
*sans* modification edward), le relire, et comparer les deux /vidages/ :

#+begin_example
edward roundtrip "dwg/EAMOA-23955/N1A_2TMV 1.dwg"   ; lit → écrit (temp) → relit → diff des dumps
#+end_example

Équivaut à =assert dump(lire(écrire(lire(F)))) ≡ dump(lire(F))=. La
commande =roundtrip= n'exerce que =read-drawing= / =write-drawing= de
clautolisp (et libredwg pour le DWG) — *pas* la logique de transfert v2 —
ce qui isole la fidélité du format de la logique d'edward.

Critères de réussite (dump avant ≡ dump après) :

1. *Mêmes appids*, même =entity_count=, mêmes handles d'entités.
2. *Toutes les xdata* préservées (par appid, octet pour octet sur la
   forme =raw=) sur chaque entité.
3. *Tous les dictionnaires nommés et XRECORD* préservés (clés, ordre,
   codes DXF, valeurs) — en particulier =SCHMS_LIGNES/VOIES/POSTES=.
4. Variables d'en-tête, blocs, calques/linetypes inchangés.

Si le round-trip diverge, on /caractérise/ la perte (quels codes, quels
objets) et on la documente comme limite de libredwg *avant* de bâtir la
v2 : les garanties du §5.2 en dépendent directement. À exécuter dès que
le jalon 0 sait lire ces deux DWG.

* Version 2 — sélection et transfert entre dessins

** Modèle d'opération

Un dessin *destination* et un ou plusieurs dessins *sources*. On
/sélectionne/ des *objets applicatifs* (instances) dans les sources,
puis on applique une *opération* vers la destination.

*Translation horizontale, sans mise à niveau (politique centrale).*
edward *n'effectue aucune migration* d'objet entre versions de schéma
(cela exigerait d'intégrer le code de migration de SCHMS,
=mise_a_jour.lsp=). Il *transfère chaque objet à sa version d'origine*,
verbatim, d'un dessin à l'autre — translation /horizontale/. C'est
ensuite *l'application SCHMS qui mettra à niveau* les objets obsolètes,
avec sa logique métier. Conséquences :

- la =schema_version= de chaque instance est *préservée à l'identique* ;
- =merge=/=replace= apparient les objets par *classe + clé/UUID*, et
  *signalent* (sans réconcilier) toute différence de version entre source
  et destination (option =--allow-version-mismatch= pour forcer) ;
- les objets divergents (cf. §3, « Principe directeur ») sont transférés tels quels, brut
  compris ; edward ne « répare » rien.

*** Sélection

Critères combinables (ET/OU) :
- par *application* / *appid* ;
- par *classe* (SCHMS : =/es/schms/sx2/gare= ; PV : la classe implicite) ;
- par *niveau* : données d'entité vs données de dessin (dictionnaires) ;
- par *valeur de champ* (ex. =UUID= = …, =NOM= contient …) ;
- par *entité support* : type, calque, plage de handles, fenêtre
  géométrique.

*** Opérations

- =merge= : ajouter les objets sélectionnés à la destination (sans
  toucher aux existants). Gestion des collisions de clé (UUID, clé de
  dictionnaire) : =--on-conflict skip|rename|replace=.
- =replace= : remplacer dans la destination les objets correspondants
  (même classe + clé/UUID) par ceux de la source. *Par défaut, les objets
  de la destination sans correspondant dans la source sont conservés.*
- =replace --prune= (= /miroir/ ou /synchronisation/) : faire en sorte
  que l'ensemble sélectionné de la destination devienne *identique* à
  celui de la source — écraser les correspondants, ajouter les manquants,
  *et supprimer les objets de la destination absents de la source*. C'est
  l'opération de *restauration intégrale* d'un dictionnaire/table : utile
  quand la destination est *corrompue* (et pas seulement incomplète), car
  les entrées corrompues à clé cassée/étrangère, qu'un =replace= par clé
  laisserait subsister, sont éliminées. Granularité de l'élagage bornée
  par la sélection (=--prune= n'efface jamais hors du périmètre
  sélectionné : p.ex. =level=drawing class=…ligne,…voie,…poste= n'élague
  que ces trois dictionnaires). Équivaut à =delete= (sur la sélection
  destination) suivi de =merge= (depuis la source), en une opération
  atomique et prévisualisable.
- =delete= : supprimer de la destination les objets sélectionnés (par
  critère), xdata et/ou XRECORD.
- =diff= : comparer source/destination sans écrire (rapport JSON :
  objets seulement-source / seulement-destination / communs-différents /
  communs-identiques, par classe et clé).

*** Entités attachées

Option transversale =--with-entities | --data-only= :
- *avec entités* : copier l'entité support (INSERT/bloc, sa définition de
  bloc si absente, ses calques/linetypes) /et/ ses xdata ; ré-allouer les
  handles dans la destination et réécrire les références internes (champs
  =HANDLE=, liens de dictionnaire).
- *données seules* : ne transférer que la charge applicative
  (xdata/XRECORD), en la rattachant à une entité existante de la
  destination (appariement par UUID/handle fourni) ou à un objet de
  dictionnaire.

*** Intégrité

- *Ré-allocation de handles* via =allocate-handle= ; table de
  correspondance =handle source → handle destination= appliquée à toutes
  les références (=1005=, =330/350/360=, champs =HANDLE=).
- *Dépendances de blocs/calques/linetypes* : copiées si manquantes.
- *Versions de schéma* : edward /préserve/ la version d'origine ;
  signalement si la destination utilise une version différente (pas de
  migration automatique en v2 ; option =--allow-version-mismatch=).
- *Sécurité d'écriture* : jamais d'écrasement en place sans =--in-place= ;
  par défaut, écriture dans un fichier de sortie distinct, et =--backup=
  pour conserver l'original. DWG réécrit via libredwg (=drawing-dwg=).

** Cas de validation V2 — restauration des tables LIGNES/VOIES/POSTES

Ce cas réel est le *test d'acceptation de la v2*. Une utilisatrice a
*perdu/corrompu* les tables /lignes/, /voies/, /postes/ d'un schéma ; elle
veut les *récupérer* depuis un schéma d'origine intact, pour que le
document altéré « retrouve les mêmes libellés ».

- Source (donnée correcte) : =dwg/EAMOA-23955/N1A 1.DWG=
- Destination (donnée corrompue) : =dwg/EAMOA-23955/N1A_2TMV 1.dwg=
- Résultat attendu : les dictionnaires =SCHMS_LIGNES=, =SCHMS_VOIES=,
  =SCHMS_POSTES= de la destination redeviennent identiques à ceux de la
  source ; les entités graphiques de la destination (variante /2TMV/) ne
  sont *pas* modifiées.

Pourquoi le cas est couvert par la spec :

- Les tables sont des *données de niveau dessin* (XRECORD dans des
  dictionnaires nommés sous le NOD, classes =/es/schms/bd/{ligne,voie,poste}=,
  catégorie « Non graphique » — §2.2) ⇒ transfert =--data-only= : on ne
  touche *pas* aux entités de la variante 2TMV.
- La donnée étant *corrompue* (et pas seulement absente), un =replace= par
  clé laisserait subsister d'éventuelles entrées à clé cassée. On veut le
  *miroir* : =replace --prune= (restauration intégrale des trois
  dictionnaires).
- Transfert *horizontal* : chaque objet BD est recopié à *sa version de
  schéma d'origine* ; SCHMS mettra à niveau ensuite si besoin (§5).

Session interactive :

#+begin_example
edward> open  src  "dwg/EAMOA-23955/N1A 1.DWG"
edward> dest        "dwg/EAMOA-23955/N1A_2TMV 1.dwg"
edward> diff  src dest  app=schms level=drawing      ; constater l'écart (recommandé)
edward> select src  app=schms level=drawing \
              class=/es/schms/bd/ligne,/es/schms/bd/voie,/es/schms/bd/poste
        -> N objets sélectionnés (lignes + voies + postes)
edward> preview replace --data-only --prune          ; rapport : à écraser / ajouter / élaguer
edward> apply
edward> save                                          ; --backup conserve l'original
#+end_example

Équivalent CLI (scriptable) :

#+begin_example
edward copy --from "dwg/EAMOA-23955/N1A 1.DWG" \
            --to   "dwg/EAMOA-23955/N1A_2TMV 1.dwg" \
            --select 'app=schms level=drawing' \
            --op replace --data-only --prune \
            -o "dwg/EAMOA-23955/N1A_2TMV 1-restaure.dwg" --backup
#+end_example

Critères de réussite (vérifiables par =edward dump= / =edward diff= avant
et après) :

1. Après opération, =diff src dest app=schms level=drawing= ne signale
   *aucune* différence sur les trois dictionnaires BD (mêmes clés, mêmes
   champs, mêmes valeurs, mêmes versions).
2. Le nombre et les handles des *entités graphiques* de la destination
   sont *inchangés* (=--data-only= n'a touché qu'aux dictionnaires).
3. Round-trip DWG fidèle : la destination réécrite se relit sans perte
   (entités + autres données applicatives intactes).

Réserves à exposer à l'utilisatrice (issues du =diff= préalable) :

- *Alignement des tags* : les libellés ne réapparaissent que si les
  entités de la destination référencent les mêmes =TAGLIGNE/TAGVOIE/…= que
  les tables restaurées. La variante /2TMV/ étant une modification, le
  =diff= révèle les tags orphelins (présents en source, absents/renommés
  côté entités destination) ⇒ récupération éventuellement *partielle*,
  signalée et non masquée.
- *Rafraîchissement SCHMS* : edward restaure la /donnée/ ; le réaffichage
  des libellés relève d'un /regen/ dans SCHMS (edward ne fait pas de CAO).

* Interface utilisateur

** Mode CLI (opérations simples, scriptables) — =edward=

Sur le modèle de =dwg-identify= (analyse d'arguments maison, =uiop:quit=
avec statut Unix : 0 ok, 1 échec fichier, 2 erreur d'usage).

#+begin_example
usage: edward [GLOBAL] <commande> [OPTIONS] [FICHIERS...]

Commandes :
  dump        vider les données applicatives + entités en JSON (v1)
  list        lister les applications/appids/classes présentes (résumé)
  roundtrip   lire→réécrire→relire et vérifier l'absence de perte (v1)
  diff        comparer deux dessins (rapport JSON)            (v2)
  copy        transférer des objets entre dessins             (v2)
  shell       ouvrir le mode interactif                        (v2)

Options globales :
  -h, --help            aide
  -V, --version         version
  -v, --verbose         détailler
      --json / --pretty / --compact   format de sortie
  -o, --output FICHIER  rediriger la sortie
      --schema-root DIR racine des *_ATTR.LSP SCHMS (déf. : arbre schms/
                        ou copie embarquée) ; --no-schema pour brut seul

dump :
  edward dump [--app A,B] [--raw] [--no-entities|--entities-only]
              [--type T] [--layer L] FICHIER.dwg ...

copy (v2) :
  edward copy --from SRC.dwg --to DST.dwg
              --select 'app=schms class=/es/schms/sx2/gare'
              --op merge|replace|delete
              [--prune]                 ; replace miroir : la sélection de DST
                                        ;   devient identique à celle de SRC
              [--with-entities|--data-only]
              [--on-conflict skip|rename|replace]
              [--allow-version-mismatch]
              [-o OUT.dwg] [--in-place] [--backup]
#+end_example

Exemples :
#+begin_example
edward dump N1A_V1.dwg > N1A_V1.json
edward dump --app pv --raw plan.dwg
edward list *.dwg
edward copy --from a.dwg --to b.dwg \
       --select 'app=schms class=/es/schms/bd/voie' \
       --op merge --with-entities -o b-merged.dwg
#+end_example

** Mode interactif (opérations complexes, multi-dessins) — =edward shell=

REPL minimal (boucle lire/évaluer/afficher) pour piloter plusieurs
sources et une destination, construire des sélections, prévisualiser
puis appliquer. État : dessins ouverts (alias), sélection courante,
destination courante.

#+begin_example
edward> open src1 a.dwg            ; ouvre a.dwg sous l'alias src1
edward> open src2 b.dwg
edward> dest c.dwg                 ; fixe la destination
edward> apps src1                  ; applications/appids présents
edward> select src1 app=schms class=/es/schms/bd/voie
        -> 128 objets sélectionnés
edward> show 0..5                  ; aperçu des objets sélectionnés (JSON)
edward> filter NOM~="V12"          ; affine la sélection courante
edward> preview merge --with-entities   ; simulation, rapport de conflits
edward> apply                      ; exécute vers la destination (en mémoire)
edward> save                       ; écrit la destination (avec --backup)
edward> quit
#+end_example

Principes : toute opération est /simulable/ (=preview=) avant =apply= ;
=apply= modifie le dessin en mémoire ; =save= seul écrit sur disque.
Annulation par rechargement de la destination tant que non sauvegardée.

* Version 3 — explorateur interactif, requêtes et validation

Réalise (et précise) le « Mode interactif » ci-dessus sous la forme d'un
/REPL Lisp/ : plutôt qu'un mini-langage de commandes =open/select/apply=, on
expose le dessin en mémoire et une bibliothèque de combinateurs, et la
« langue de requête » /est/ le langage des S-expressions. But premier :
inspecter les dessins produits par libredwg pour /débusquer les problèmes de
compatibilité R2018 restants/, et offrir un contrôle de normes façon
application standard de BricsCAD.

** Modèle de nœud — adressage uniforme à tous les niveaux

Un =node= étiquette un /niveau/, la valeur sous-jacente (/payload/), le nœud
parent, le dessin propriétaire et l'index/clé dans le parent. Niveaux :
=:drawing :entity :object :table-record :dict :dict-entry :header-var :pair
:block :xdata-group :instance :field=. Chaque niveau est produit par une
fonction existante (=entity-xdata=, =walk-dictionaries=, =decode-instance-stream=,
le lecteur d'accolades =%read-value=…) : l'explorateur /navigue et présente/,
il ne reparse rien. =mutation-target= remonte au plus proche =:entity= /
=:object= (seules unités modifiables : =modify-entity= remplace la liste DXF
entière).

** Langage de requête (=query.lisp=)

Style combinateur, =sélection -> sélection= (une sélection est une simple
liste de nœuds, donc toute la bibliothèque de séquences CL compose) :
- racines : =(entities d)= =(objects d)= =(table-records d)= =(dictionaries d)=
  =(header-vars d)= ;
- navigateurs : =children= =descendants= =parent-of= =ascend-to= ;
- filtres (tous via =where=) : =by-layer= =by-kind= =by-appid= =by-class=
  =by-code= =by-version= =by-handle= =by-level= =pick= =first-of= ;
- terminaux : =count-of= =to-list= =show= =export-sel= ;
- fil : =(chain d (entities) (by-layer "SIGNAUX") (by-appid "SCHMSPLUS")
  descendants)=.

** REPL et exécution non interactive

Trois sous-commandes ajoutées au CLI :
#+begin_example
edward repl [FICHIER]        ; REPL Lisp interactif (*drawing* / *sel* liés)
edward eval 'FORME' [FICHIER]; évalue une forme et affiche le résultat
edward script [FICHIER]      ; évalue les formes lues sur stdin
#+end_example
La boucle s'exécute dans le paquet =edward= (vocabulaire sans préfixe) ; les
nœuds et sélections s'affichent en /arbre/ (=show=, =examine=), les autres
valeurs avec l'imprimeur Lisp normal. =(help-repl)= rappelle le vocabulaire.
=rlwrap edward repl f.dwg= pour l'édition de ligne (aucune dépendance ajoutée).

** Inspection, visualisation, export (=inspect.lisp=)

=show= / =print-tree= rendent un nœud (ou une sélection) en arbre ASCII avec
en-têtes par niveau ; =examine= déplie en profondeur et indique la cible de
mutation. =export-sel= sérialise toute sélection en =:json= (réutilise les
émetteurs de =dump.lisp=), =:sexp= (Lisp relisible) ou =:dxf-fragment=.

** Éditeur structuré de S-expressions (=sedit.lisp=)

Les données d'edward /sont/ des arbres de conses, donc l'édition passe par un
éditeur structuré à curseur, adapté de =small-cl-pgms/sedit= (AGPL-3.0, même
auteur). Commandes : navigation =d/u/f/b= (descendre/monter/avant/arrière),
édition =i/a/r/x/c/y= (insérer/ajouter/remplacer/couper/copier/coller). =(edit
node)= édite l'arbre de la cible de mutation puis réécrit via le point de
passage unique =edit-entity-pairs= → =modify-entity= / =add-object=.

** Opérations (=ops.lisp=) — écriture en DXF uniquement

=del= (effacer entités/objets/entrées de dico), =dup= (dupliquer avec
handles frais), =ins= (insérer une entité), =edit= (édition structurée),
=save= (=.dxf= UTF-8). L'écriture DWG native est refusée tant que l'encodeur
R2018 de libredwg est indisponible ; le flux reste : produire un DXF, le
rouvrir dans le CAD, /Enregistrer sous/ DWG.

** Moteur de validation (=validate.lisp=) — détecteur de bugs / normes CAO

Une /règle/ est une donnée : =id=, =level=, =severity=, =description=,
=predicate= (nœud → =nil= ou message). =defrule= les enregistre dans =*rules*=
(l'utilisateur ajoute ses règles métier SNCF sans toucher au moteur — analogue
des normes CAO). =run-rules= produit un rapport (texte ou JSON :
=edward validate FICHIER [--json] [--schema-root DIR]=). Jeu de départ, qui
double comme détecteur de régressions libredwg :
- =:garbage-double= (=error=) — double NaN/Inf/dénormal/proche de l'overflow
  (corruption de décompression ; ±1d20 = sentinelle AutoCAD, toléré) ;
- =:xdata-grammar= / =:xrecord-grammar= — flux d'instance SCHMS non conforme ;
- =:schema-divergence= — instance violant son =*_ATTR.LSP= (si =--schema-root=) ;
- =:value-range= — code booléen (290–299) hors {0,1} ;
- =:layer-color= — couleur de calque hors ACI 1–255 (signe = on/off) ;
- =:entity-count-drift= (=error=) — perte d'entités sur un aller-retour DXF ;
- =:dangling-handle= (=info=) — référence vers un handle non modélisé
  (clautolisp ne retient pas tous les objets de contrôle : sur un fichier sain
  c'est surtout du bruit structurel ; le /signal/ est l'écart par rapport à un
  dessin de référence sain).

* Risques et questions ouvertes

- *SCHME/SCHMIEUX — codec inconnu.* /Décidé/ : forme brute pour
  l'instant (sans perte) ; décodeur dédié *affiné plus tard*, sur relevé
  de dessins réels (jalon 5).
- *Maturité de l'API d'écriture clautolisp.* La couche d'écriture est
  récente ; absence de mutateurs xdata/XRECORD de haut niveau. /Mitigation/ :
  edward construit les listes DXF et isole tout derrière =dwg-access=.
  Vérifier tôt (jalon 3) le comportement réel de =add-object=,
  =dictionary-put=, =modify-entity= et du round-trip DWG via libredwg.
- *Round-trip DWG — deux causes distinctes (analyse jalon 1, 2026-06).*
  1. /Encodage — CORRIGÉ dans clautolisp (v1.2.7)./ La chaîne DWG↔DXF
     passe par libredwg, qui transcode les chaînes en *UTF-8* et inscrit
     =$DWGCODEPAGE ANSI_1252=. clautolisp lisait/écrivait cet échange en
     ISO-8859-1 et ignorait le code page (=drawing-codepage= nil) ⇒ texte
     accentué /mojibaké/ à la lecture (=kilomètre→kilomÃ¨tre=). Correctif :
     =dxf-read-drawing=/=dxf-write-drawing= acceptent =:external-format= ; le
     codec DWG passe =:utf-8= ; le lecteur DXF capture =$DWGCODEPAGE= dans
     =drawing-codepage=. Vérifié : =edward dump= rend =kilomètre= et
     =codepage: ANSI_1252=.
  2. /Écriture DWG native — bloquée par libredwg (non corrigeable ici)./
     =dxf2dwg= n'écrit que r12/r14/r2000/r2004 ; *R2018 (=AC1032=, le format
     de ces dessins) est « Planned », non implémenté*. Rétrograder les
     attributs R2018 (ATTRIB à AcDbXrecord/MTEXT imbriqué, code DXF 43)
     vers r2000/r2004 échoue (=Invalid DXF code 43 for ATTRIB=) ou boucle.
     Reproduit en pur libredwg (=dwg2dxf | dxf2dwg=), sans clautolisp.
  *Conséquence* : la v2 émet du *DXF* (UTF-8, ouvrable et ré-enregistrable
  en DWG R2018 par BricsCAD/AutoCAD) ; l'écriture DWG native (et =--in-place=
  sur un =.dwg=) reste *bloquée* tant que libredwg n'implémente pas
  l'écriture R2018. Voir [[file:../README.md][README]].
- *Sémantique de =replace=/=delete=.* Clé d'identité d'un objet
  applicatif (UUID ? clé de dictionnaire ? handle ?) à fixer par
  application ; comportement par défaut prudent (=skip= sur conflit).
- *Schémas =*_ATTR.LSP=.* /Décidé/ : edward les lit (typage, validation,
  affichage, recherche). À fixer en implémentation : la *racine* par
  défaut (=--schema-root= pointant l'arbre =schms/= vs copie embarquée
  versionnée avec edward, plus stable mais à resynchroniser) ; le
  décodage auto-descriptif reste le repli quand un schéma manque. Les
  fichiers sont en *cp1252/CRLF* — décodage obligatoire.
- *Divergence DWG vs spec applicative.* Attendue et centrale (§3, « Principe directeur ») : des
  xdata réelles peuvent ne pas respecter les =*_ATTR.LSP=. edward la
  /révèle/ (annotations) sans la corriger. À surveiller : volume des
  divergences sur dessins réels, pour calibrer l'affichage.
