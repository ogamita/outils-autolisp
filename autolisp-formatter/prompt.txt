PROJECT GOAL
Create a deterministic AutoLISP source formatter / pretty-printer written in AutoLISP itself.

HIGH LEVEL REQUIREMENTS
- Implementation language: AutoLISP.
- It must parse AutoLISP source text without evaluating it.
- It must preserve semantics and avoid corrupting code.
- It must support two parenthesis styles:
  1. CL style: close parens on the last possible line.
  2. Nail-clipping style: closing parens may appear on separate lines in a consistent historical AutoLISP style.
- It must support symbol case modes:
  1. preserve
  2. downcase
  3. upcase
- It must support comment policies:
  1. preserve
  2. CL-inspired normalization using ;;;;, ;;;, ;;, ;
- It must understand AutoLISP block comments of the form ;| ... |;.
- It must optionally convert block comments ;| ... |; into semicolon-prefixed line comments.
- It must support check mode and in-place mode.
- It must be deterministic and idempotent.

IMPORTANT LANGUAGE NOTES
- In this project, :foo and :bar are ordinary AutoLISP symbols, not Common Lisp keywords.
- / in a defun parameter list is also just a symbol lexically.
- The formatter must distinguish symbols from strings and comments. Never alter string contents. Never alter comment text except when explicitly converting block comments.
- The parser is syntactic only. No evaluation.

SCANNER REQUIREMENTS
Tokenize at least the following:
- lparen
- rparen
- quote
- string
- atom
- line-comment
- block-comment ;| ... |;
- whitespace
- newline

Scanner must correctly ignore parentheses inside strings and comments.
Scanner must detect unterminated strings and unterminated block comments.
Scanner must accept symbols such as:
- pk!
- vl-catch-all-apply
- *debug*
- :foo
- /

PARSER REQUIREMENTS
Implement a lightweight parser that builds a CST or equivalent tree preserving enough surface syntax for pretty-printing.
It must support:
- top-level forms
- nested lists
- quote
- atoms
- strings
- comments attached as trivia or equivalent
- dotted pair forms without breaking them

FORMATTING REQUIREMENTS
Support at least these forms with better indentation rules:
- defun
- lambda
- if
- cond
- setq
- progn
- while
- repeat
- foreach
- quote

PAREN STYLE REQUIREMENTS
- style=cl
- style=nail
Make the output deterministic in both styles.

CASE REQUIREMENTS
Modes:
- preserve
- downcase
- upcase
Apply only to symbol tokens.
Do not alter strings, numbers, whitespace, or comment bodies.

COMMENT REQUIREMENTS
Modes:
- comment-style=preserve
- comment-style=cl
Support optional conversion:
- block-comments=preserve
- block-comments=to-semicolons
Support optional contextual comment conversion heuristics:
- top-level isolated comments -> ;;;; or ;;;
- local block comments -> ;;
- inline comments -> ;
Support optional inline comment alignment column.

CLI / ENTRYPOINT REQUIREMENTS

- --comment-style preserve|cl

- --inline-comment-column N

- --check
